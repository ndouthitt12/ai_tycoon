## 1. State Management and Performance (copyGame)

The game loop relies heavily on creating a deep copy of the state before applying mutations.

 - The Issue: The copyGame function uses JSON.parse(JSON.stringify(game)). While functional, this is one of the slowest methods for deep cloning in JavaScript. In a simulation game where the state object grows massive (containing historical arrays, competitor models, and cohorts), invoking this every time a player clicks a button or advances a month will eventually cause noticeable UI lag and frame drops.

 - The Solution: * Immediate fix: Use the native Web API structuredClone(game), which is significantly faster and natively handles complex data types (like Maps or Dates, should they be added later).

 - Robust fix: Introduce a library like Immer. Immer allows you to write mutating logic (e.g., draft.cash -= 500) while it handles producing a structurally shared, immutable next state under the hood. This pairs perfectly with your current architecture.

## 2. Architectural Modularity (The God File)

The sim.ts file acts as a "god file," currently stretching to nearly 1,500 lines.

 - The Issue: It mixes formatting utilities (like formatBigMemory), economic logic (settleGlobalMarket), competitor AI (updateCompetitorCompanies), and event resolving all in one place. This makes the file difficult to navigate, increases the likelihood of merge conflicts, and muddles the domain logic.

 - The Solution: Split this logic into a dedicated engine/ or systems/ directory.

   -- utils/formatters.ts: Move money, formatPods, pct, formatBigParams, etc.

   -- engine/market.ts: Move settleGlobalMarket and pricing logic.

   -- engine/competitor-ai.ts: Move updateCompetitorCompanies and release planning logic.

   -- engine/training.ts: Move calculateRunPreview, launchRun, and model generation logic.

## 3. Magic Numbers and Hardcoded Constants
Throughout the simulation engine, raw integers and floats dictate game balance.

  - The Issue: Examples include (modelAges[key] - avgAge) * 0.0022, / 4.5, and multiplying baseline costs by arbitrary multipliers like 0.38 or 0.55. These "magic numbers" make the code hard to interpret. If you need to rebalance the game's economy later, finding and adjusting these scattered multipliers will be a nightmare.

  - The Solution: Extract these coefficients into a balance.config.ts file or define them as named constants at the top of the file (e.g., const AGE_CHURN_PENALTY_COEFFICIENT = 0.0022).

## 5. Context Window / KV Cache Tax

    Currently, a 128K context window model costs the same to serve as an 8K model if they generate the same number of tokens. In reality, large context windows consume massive amounts of VRAM to store the KV Cache, reducing the batch size a pod can handle.

    - Implementation: Modify the compute demand formula to penalize high-context usage.

      -- contextPenalty = 1 + (averageContextUsed / 32) * 0.5

    - A pod's baseline throughput should plummet as the context fills up, forcing the player to build more datacenters if they want to serve massive documents.
