# Change Plan: Capability Signaling, Key Researcher Risk, and Inference Economics

This document lays out how I would implement three connected systems in the current codebase:

1. Separate real model capability from demonstrated capability.
2. Replace pure aggregate researcher power with named key researchers and key-person risk.
3. Make inference economics a first-class operational loop instead of a mostly static post-launch derivative.

The intent is to deepen the simulation without breaking the current monthly cadence or making the UI unreadable.

## Why these three changes belong together

Right now the game has a few simplifying assumptions:

- `ModelState.capability` is doing too much work. It acts as internal model quality, public benchmark standing, and market perception all at once.
- Research is represented mainly as `headcount.researchers`, so the lab has scale but not fragility.
- Serving demand is derived from token usage and averaged inference cost, but the player is not yet making many moment-to-moment decisions around burst demand, batching quality, and delayed capacity expansion.

These requested changes all push the game toward the same design principle: the best AI company is not the one with the biggest raw number, but the one that can convert talent, capability, and infrastructure into trusted, profitable public outcomes.

## Design goals

- Preserve the current monthly simulation loop in `src/game/sim.ts`.
- Keep player-facing complexity legible: each system should surface 2-4 decisions, not 20 hidden knobs.
- Avoid replacing every existing system at once. Prefer additive changes and phased migration.
- Make trust and margin more sensitive to overclaiming, researcher churn, and serving instability.

## 1. Separate capability from demonstrated capability

### Current state

Today, a model has one core quality metric:

- `ModelState.capability` in `src/game/types.ts`

That value feeds:

- training previews in `src/game/systems/training.ts`
- market comparisons in `src/game/systems/market.ts`
- competitor comparisons in `src/game/systems/competitors.ts`
- the model tables in `src/screens/LabScreen.tsx`, `src/screens/MarketScreen.tsx`, and `src/screens/OverviewScreen.tsx`

This makes the market perfectly informed and removes the possibility of selective disclosure, benchmark gaming, or delayed trust damage.

### Proposed implementation

Add a split between:

- `trueCapability`: the model's latent underlying power
- `demonstratedCapability`: what the market currently believes, based on released evals and observed product performance

I would not delete `capability` immediately. I would migrate in phases:

### Phase 1 data model

Extend `ModelState` with fields like:

```ts
trueCapability: number;
demonstratedCapability: number;
benchmarkProfile: {
  internal: Record<string, number>;
  published: Record<string, number>;
  withheld: string[];
};
capabilityCredibility: number;
```

Add a light benchmark taxonomy in `src/game/defs.ts`, for example:

- general
- reasoning
- coding
- multimodal
- safety
- latency_efficiency

The important part is not benchmark realism; it is that some benchmarks flatter certain model shapes better than others.

### Model generation logic

In `src/game/systems/training.ts` and the run-completion path in `src/game/sim.ts`:

- rename the current computed `capability` concept internally to `trueCapability`
- generate per-benchmark internal scores from:
  - true capability
  - goal mix
  - trust / reliability
  - specialization bonuses
- initialize `demonstratedCapability` from published results only, not the full internal score set

The simplest first pass:

- `trueCapability` still drives actual product quality
- `demonstratedCapability` is computed from the weighted average of published benchmark scores

### New player decision: what results to publish

Add a release choice after a model ships, likely as:

- a lightweight modal or pending event in `src/game/sim.ts`
- or a release panel in `src/screens/LabScreen.tsx`

The player chooses which benchmark categories to publish.

Publishing strong cherry-picked results gives:

- higher short-term acquisition
- higher board excitement
- stronger market-table placement

But it also raises `capabilityCredibility` risk if real-world product usage underperforms the implied claim.

### Trust and backlash loop

In `src/game/systems/market.ts`, compare:

- product outcomes driven by `trueCapability`
- market expectations driven by `demonstratedCapability`

If demonstrated capability is materially above experienced value:

- increase churn
- reduce product trust
- reduce global company trust
- add a notification about benchmark skepticism or customer disappointment

If demonstrated capability slightly trails true capability:

- acquisition is slower
- but trust can rise over time as user experience beats expectations

This creates the intended tradeoff: real capability compounds, hype decays.

### Competitor integration

Competitors in `src/game/systems/competitors.ts` should also have:

- latent strength
- public benchmark posture

This lets rivals feel more human:

- some are disciplined and under-promise
- some are benchmark-maximizers
- some are multimodal demo-heavy but weak in enterprise reliability

### UI changes

Update the market and model screens to show:

- `Demonstrated Capability`
- a public benchmark summary
- optionally a tooltip or badge for `Overperforming Expectations` / `Under Scrutiny`

Do not show `trueCapability` directly in the market screen.
Do show it in admin/debug views or possibly only in the lab for the player's own internal models.

### Files likely touched

- `src/game/types.ts`
- `src/game/defs.ts`
- `src/game/systems/training.ts`
- `src/game/systems/market.ts`
- `src/game/systems/competitors.ts`
- `src/game/sim.ts`
- `src/screens/LabScreen.tsx`
- `src/screens/MarketScreen.tsx`
- `src/screens/AdminScreen.tsx`

## 2. Key person risk for researchers

### Current state

Research output is mostly aggregated through:

- `game.headcount.researchers`
- `getResearchContribution()` in `src/game/systems/training.ts`

This gives scale but not specialization, staffing decisions, or lab fragility. The game design doc already points toward richer talent dynamics, but the current implementation is still coarse.

### Proposed implementation

Keep aggregate researcher headcount for baseline throughput, but layer a named researcher roster on top.

This should be additive, not a full org-sim rewrite.

### Phase 1 data model

Add a `researchers` array to `GameState`, with a new type such as:

```ts
interface ResearcherState {
  id: number;
  name: string;
  specialty: "reasoning" | "multimodal" | "safety" | "inference" | "agentic" | "data";
  seniority: "staff" | "principal" | "distinguished";
  skill: number;
  loyalty: number;
  burnout: number;
  poachRisk: number;
  breakthroughChance: number;
  assignedRunId: number | null;
  active: boolean;
}
```

Important: this does not replace all researchers. It represents only the named, high-leverage people.

Interpretation:

- `headcount.researchers` remains broad team mass
- `researchers[]` represents the key individuals whose presence can swing a run

### Starting state and hiring

In `src/game/defs.ts` or a new small generator module:

- seed each archetype with 1-3 named researchers
- bias frontier labs toward reasoning / safety stars
- bias consumer firms toward multimodal / product-facing researchers
- bias enterprise firms toward safety / reliability / inference researchers

Later, hiring can recruit named researchers through events instead of a full staffing marketplace.

### Run staffing

Extend `ActiveRun` in `src/game/types.ts` with:

- `assignedResearcherIds: number[]`
- `specializationBonuses`
- `keyPersonRisk`

In `src/game/systems/training.ts`:

- keep the baseline headcount contribution
- add specialty modifiers from assigned named researchers

Examples:

- reasoning researcher boosts reasoning benchmark generation and reliability in analytical tiers
- multimodal researcher boosts multimodal benchmark strength
- safety researcher lowers trust downside and failure spillover
- inference researcher slightly reduces projected inference cost or post-training serving weight

This creates a real staffing decision instead of a passive scalar.

### Mid-run risk events

Add monthly event checks in `advanceMonth()` inside `src/game/sim.ts` for assigned named researchers:

- poaching attempt
- burnout spike
- breakthrough
- internal conflict / reassignment request

Losing a researcher mid-run should create a real decision point:

- continue and accept reduced outcome
- pause the run and replace them
- spend cash/equity to retain them
- redirect the run to a safer objective

This maps cleanly onto the existing pending-event system.

### Breakthroughs

A breakthrough should not be just a random free bonus. It should respect specialty:

- reasoning breakthrough: bonus to true capability, reasoning evals, or reliability tiers
- multimodal breakthrough: bonus to multimodal internal scores and acquisition appeal
- safety breakthrough: trust bonus and lower downside from hype mismatch
- inference breakthrough: lower serving cost and better batching efficiency

This also helps tie the talent system directly into the inference-economics loop.

### UI changes

On the lab screen:

- add a small `Key Researchers` panel
- show specialty, burnout, loyalty, and current assignment
- allow assigning researchers to runs before launch

On active runs:

- show key assigned researchers as badges
- warn when a run is exposed to single-person failure

### Files likely touched

- `src/game/types.ts`
- `src/game/defs.ts`
- `src/game/systems/training.ts`
- `src/game/sim.ts`
- `src/screens/LabScreen.tsx`
- `src/screens/ComputeScreen.tsx` for inference researchers if surfaced there

## 3. Inference economics as a core loop

### Current state

The project already has a good base:

- context overhead exists in `src/game/systems/market.ts`
- serving demand is visible in `src/screens/ComputeScreen.tsx`
- overflow cost matters in `src/game/sim.ts`

But serving economics are still mostly:

- average token demand
- average inference cost
- static reserved capacity allocation

The requested change is to make serving economics feel like the central operational tension during growth.

### Proposed implementation

Split serving economics into three layers:

1. Demand generation
2. Effective throughput
3. Capacity response lag

### A. Demand generation

Keep token usage, but make it more bursty and more sensitive to success.

Add to `ProductState`:

```ts
traffic: {
  baselineTokensMillions: number;
  burstMultiplier: number;
  averageContextUtilization: number;
  batchingFriendliness: number;
  viralPressure: number;
}
```

The key change:

- a strong launch or competitor shock can create temporary demand spikes
- serving costs rise immediately
- capacity expansion only arrives after a delay

This creates the exact tension the request calls out.

### B. Effective throughput model

Replace the current mostly linear compute usage calculation in `getProductComputeUsage()` with something closer to:

```ts
effectiveTokensPerPod =
  baseTokensPerPod
  * batchingEfficiency
  / contextPenalty
  / modelWeightPenalty
  / hardwareMismatchPenalty;
```

Core drivers:

- context length or utilization
- batching efficiency
- hardware quality / cloud upgrade level
- model inference weight
- routing policy

This means cost per token is not a fixed property of the model alone.

### C. Capacity response lag

Datacenter builds already cost money, but they are too immediate to create true operational panic.

Introduce:

- provisioning lead time for datacenters
- optional short-term cloud burst rental at punitive rates
- temporary batch degradation when traffic spikes beyond healthy throughput

Implementation idea:

- new datacenters enter `plannedDatacenters` first
- they become active after `n` months
- overflow can be covered by `burstCloudPods` at high cost before permanent capacity arrives

This makes virality dangerous in a fun way instead of always good.

### D. Product routing strategy

Add a small serving strategy selector per product:

- flagship only
- tiered routing
- aggressive cache / cheap routing
- enterprise SLA mode

These should influence:

- quality experienced by users
- cost per token
- trust
- churn under pressure

This can be implemented incrementally without introducing multiple live models per request path at full realism.

### E. Margin and trust consequences

Serving failure should do more than charge overflow cost.

If the player goes viral without capacity:

- gross margin collapses
- trust falls faster
- enterprise users churn harder than consumer users
- board pressure rises because the company is buying growth at negative unit economics

This should be reflected in:

- `lastMonth`
- product trust
- company trust
- board pressure

### F. Competitor behavior

Competitors should also express a serving posture:

- premium but capacity constrained
- cheap and efficient
- enterprise stable
- hype-driven and outage-prone

This will make the market feel more believable and lets the player win with operational excellence rather than only top-end capability.

### UI changes

Enhance `src/screens/ComputeScreen.tsx` with:

- average context utilization
- batching efficiency
- effective tokens per pod
- burst cloud usage
- queued capacity / datacenter lead times
- serving strategy per product

Enhance overview/product areas to show:

- gross margin pressure
- traffic spikes
- why compute demand jumped this month

### Files likely touched

- `src/game/types.ts`
- `src/game/systems/market.ts`
- `src/game/sim.ts`
- `src/game/defs.ts`
- `src/screens/ComputeScreen.tsx`
- `src/screens/OverviewScreen.tsx`

## Recommended implementation order

I would implement this in four passes instead of trying to land all three systems at once.

### Pass 1: Data model and hidden/public capability split

Goal:

- add `trueCapability` and `demonstratedCapability`
- preserve current balance as much as possible
- swap market-facing reads to demonstrated capability

Why first:

- this is the least invasive conceptual change
- it creates the foundation for hype, trust decay, and researcher specialization

### Pass 2: Named researchers and run staffing

Goal:

- add key researchers
- assign them to runs
- make breakthroughs / poaching / burnout event-driven

Why second:

- once models have richer internal profiles, specialized researchers have a meaningful place to land

### Pass 3: Dynamic serving economics

Goal:

- introduce burst demand, throughput efficiency, and capacity lag
- make compute screen more operationally rich

Why third:

- it is the broadest economy rebalance and will be easier to tune once the capability and talent systems are stable

### Pass 4: Competitor parity and polish

Goal:

- give competitors the same public/private capability split
- add rival researcher events and serving styles
- polish market UI and admin/debug views

Why fourth:

- competitors are multiplicative complexity; player systems should feel right first


## Balance and tuning notes

These systems can go wrong if they are too punishing or too noisy.

Key guardrails:

- Cherry-picking benchmarks should help enough that players are tempted.
- Trust backlash should lag by a few months so it feels like decay, not instant punishment.
- Named researcher events should be rare but meaningful. They should feel like board-level moments, not monthly spam.
- Viral inference spikes should threaten margins before they threaten immediate game over.
- Enterprise customers should care more about credibility and serving stability than consumers do.

## Testing strategy

I would validate this in layers.

### Capability split

- training a model with no published benchmarks should keep demonstrated capability muted
- publishing only favored benchmarks should raise demonstrated capability above experienced quality
- repeated underdelivery should reduce trust and acquisition

### Researcher risk

- assigning a reasoning researcher should improve reasoning-facing outcomes
- losing an assigned researcher mid-run should alter projected results and surface a decision
- burnout should accumulate under repeated staffing pressure

### Inference economics

- longer context and lower batching friendliness should reduce effective throughput
- a viral month should spike serving cost before new capacity comes online
- burst demand should meaningfully affect overflow, trust, and board pressure

## Biggest implementation risks

- If both `capability` and `trueCapability` exist too long, the code will drift into confusion. Migration should be deliberate.
- Named researchers can become busywork if assignment is mandatory for every tiny run.
- Serving economics can become opaque if the UI does not explain why costs changed.
- Competitor parity can wait; forcing it too early will slow the player-facing systems.

## Recommended MVP scope

If we want the strongest return for the least code, the MVP should be:

1. Add `trueCapability` and `demonstratedCapability`.
2. Add benchmark publication choices plus trust penalties for overclaiming.
3. Add 3-5 named researchers total, each with one specialty and simple assignment.
4. Add burst traffic, batching efficiency, and datacenter provisioning delay.

That version would already create the intended fantasy:

- you can build a genuinely strong model
- you can decide how much to brag about it
- your star reasoning researcher can leave at the worst possible time
- your best launch can become a margin crisis if serving collapses

## Concrete next step

If I were implementing this next, I would start by changing the type layer in `src/game/types.ts`, then update training/model creation so every newly shipped model has both hidden and public capability fields before touching the UI.
