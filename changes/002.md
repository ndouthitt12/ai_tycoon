**Executive Summary**
The current imbalance between Chatbot and API token consumption stems from hardcoded discrepancies in your compute demand formulas and an improper scaling of tokens based on context window size. Currently, the API product calculates cloud pod demand using a divisor of `4.5`, while the Chatbot uses `50`, meaning the API artificially demands ~11x more server capacity per token. Additionally, the context window size directly multiplies token counts, creating exponential inflation rather than representing memory (VRAM) overhead. 

To resolve this, you need to standardize the token-to-pod baseline, decouple token generation from context overhead, and introduce cohort-based API consumption rather than generic multiplier caps.

Here is a detailed breakdown of the specific problems in your code and how to fix them.

### 1. The Pod Divisor Discrepancy
In `sim.ts`, the `getProductComputeUsage` function treats Chatbot and API compute entirely differently. 

For the Chatbot, you define:
`let demand = Math.ceil(product.tokenUsageMillions / 50);` (Meaning 1 Pod = 50M tokens).

For the API, you define:
`const demand = (totalTokensMillions * metrics.inferenceCost) / 4.5;` (Meaning 1 Pod = 4.5M tokens). 

Because the API divides by 4.5, it requires over 10 times the number of cloud pods to process the same amount of tokens. 

**The Fix:** Standardize the hardware baseline. A token is a token. The difference in cost should come strictly from the model's `inferenceCost` and context utilization, not a hidden product divisor. 

```typescript
// Proposed unified compute logic in getProductComputeUsage
const baseTokensPerPod = 50; // 50M tokens = 1 Pod

let totalTokensMillions = 0;
if (product.type === "chatbot") {
  totalTokensMillions = product.tokenUsageMillions;
} else {
  // API token generation logic
  totalTokensMillions = calculateApiTokens(game, product, metrics);
}

// Both products now use the same baseline hardware capability
let baseDemand = totalTokensMillions / baseTokensPerPod;
```

### 2. The Context Window "Token Inflation" Bug
In `settleGlobalMarket`, you currently have this logic for calculating token usage:
```typescript
const cWindowScale = Math.max(1, contender.model.contextWindow / 8);
const contextScaledTokensM = (usedTokensM || 0) * cWindowScale;
```
This implies that if a user uses a 128k context model to say "Hello", they generate 16x more tokens than if they used an 8k model. This causes the "millions of millions of tokens" explosion because context window size is directly multiplying the raw token throughput.

**The Fix:** A large context window doesn't mean the user *generates* more tokens; it means the server uses more VRAM (KV Cache) to hold the conversation open. You should remove `cWindowScale` from the token calculation entirely and apply a "Context Overhead Tax" strictly to the *pod demand*.

```typescript
// In settleGlobalMarket:
// Remove cWindowScale. Just use the actual tokens.
const actualTokensM = usedTokensM || 0; 
plan.tokenUsageMillions += usersForPlan * actualTokensM;

// In getProductComputeUsage:
// Apply a context tax to the hardware demand, not the token count.
// Example: Every 32k over the base 8k adds 25% pod overhead due to memory constraints.
const contextOverhead = 1 + Math.max(0, (metrics.contextWindow - 8) / 32) * 0.25;

return {
  demand: baseDemand * metrics.inferenceCost * contextOverhead,
  tokensMillions: totalTokensMillions,
};
```

### 3. Rebalancing API Token Consumption
Right now, `tokensPerClientMillions` scales between `0.35` and `3.5` based on a linear combination of trust, market advantage, and recency (`clamp(1 + ... , 0.35, 3.5)`). 

If `activeUsers` for the API is 100,000 clients, hitting the 3.5M cap means generating 350 Billion tokens instantly. This lacks nuance between a small startup client and an enterprise client.

**The Fix:** Tie API consumption strictly to the `GlobalCohorts` just like the chatbot, rather than a generic multiplier. 
1. Create a `Business` cohort property called `baseTokensPerMonth`. (e.g., "Tech" cohort = 50M tokens/month, "Retail" = 5M tokens/month).
2. When resolving market share in `settleGlobalMarket`, calculate the API tokens based on the cohort the users came from.
3. Use the `trust` and `capability` metrics to increase the *number of users* acquired, rather than magically making each user consume more tokens.

### 4. Increasing Chatbot Token Realism
Chatbot token usage feels low because the current math relies on `cohort.sessionsPerMonth * cohort.tokensPerSession`. If a casual user has 10 sessions of 500 tokens, that's only 5,000 tokens per month.

**The Fix:** Factor in "System Prompts" and "Conversation History". When a user chats, the LLM doesn't just read the new prompt; it re-reads the entire previous conversation.
You can simulate this by applying a conversation depth multiplier.

```typescript
// Instead of raw sessions * tokens:
const baseSessionTokens = cohort.sessionsPerMonth * cohort.tokensPerSession;

// Power users have longer conversation threads, meaning the server has to re-process history.
const conversationDepthMultiplier = cohort.id === "power" || cohort.id === "developer" ? 3.5 : 1.5;

const neededTokensM = (baseSessionTokens * conversationDepthMultiplier) / 1000000;
```

This will naturally lift chatbot token consumption to realistic levels without artificially penalizing the player's cloud infrastructure.
